# 에이전트 역할 정리

Pi 기반 행사 기획 AI 에이전트 서비스의 에이전트 구성. **메인(`event-organizer-main`=Planner)은 세션 메인 프롬프트 `AGENTS.md`**(서브에이전트가 아니라 세션 자체의 얼굴), 서브에이전트 5종 정의 원본은 `.pi/agents/*.md`이며, 이 문서는 각 에이전트의 **역할·경계·연동 관계**를 요약한다.

**불변 원칙:** 행사 1개 = 워크스페이스 1개 = 작업폴더(`cwd`) 1개 = Pi 세션 1개 = 결과 리포트 1개. 한 세션 안에서 아래 에이전트들이 일한다. 홈 화면은 세션이 아니라 여러 세션을 가로지르는 조회 대시보드이며 **에이전트가 없다.**

> 호칭은 **파일명 기준**으로 통일한다. 프롬프트 본문에는 다른 호칭(`writer`→"Author", `monitor`→"Watcher")이 쓰인 곳이 있으나, 이 문서에서는 파일명을 정본으로 본다.

| 파일 | 호칭 | 한 줄 역할 |
|---|---|---|
| `AGENTS.md` *(세션 메인)* | **event-organizer-main** | 기획 메인 엔진. 조사·작성·검토를 위임·종합해 제안서 완성, 메일 본문 준비 |
| `researcher.md` | **researcher** | 장소·업체·시세·사례·날씨 등 외부 정보 조사 (일회성, 출처 필수) |
| `writer.md` | **writer** | 제공된 데이터로 제안서 초안 작성 (본문 호칭 "Author") |
| `critic.md` | **critic** | 제안서 다각도 검토 후 문제 지적·통과 판정 (직접 수정 안 함) |
| `secretary.md` | **secretary** | 외부 통신(이메일 등) 메시지를 읽어 회의록으로 정리 |
| `monitor.md` | **monitor** | 워크스페이스 변화를 감지해 재기획 필요성 판단, main에 깃발 올림 (본문 호칭 "Watcher") |

---

## event-organizer-main — 기획 메인 엔진 (`AGENTS.md` = 세션 메인 프롬프트)
- **위치:** 워크스페이스(=한 행사 = 한 Pi 세션)당 하나. **서브에이전트가 아니라 세션의 메인** — `Agent`로 spawn되지 않고, 세션 시작 시 `AGENTS.md`로 로드되어 사용자와 직접 대화하는 얼굴이 된다. 전체를 조율하는 오케스트레이터가 아니라 **한 행사의 기획 한 바퀴를 책임지는 메인 엔진**이며, 자기 행사 하나만 책임진다(홈·다른 행사는 모름).
- **직접 수행:** 사용자 조건 접수 → 부족하면 추가 정보 요청(전 과정에 걸쳐) → 조사(researcher 병렬)·작성(writer)·검토(critic) 위임 결과 종합 → 제안서 완성. 과거 사례 DB는 `case_search`로 **직접 조회**. 메일 본문(초청·공지 등) 준비. monitor의 재기획 깃발을 받아 **영향받는 부분만** 재기획·갱신.
- **위임 방식:** 조사는 **병렬 fan-out**(researcher 다중 인스턴스, 각 `run_in_background`), 작성·검토는 **writer ⇄ critic 순차 루프**(최대 2회). 상세는 §"오케스트레이션".
- **위임/비수행:** 실제 메일 발송(승인 게이트 거쳐 Gmail MCP), 정산·적립 확정(계산만 직접, 데이터 사용은 승인 게이트), 상태 변경(`update_state` 통로만), 변화 감지(monitor), 통신 해석(secretary).
- **통제 철학:** 초안·조사·계산은 AI가 빠르게, **확정·수정·발송은 사람이 개입**. 돈·외부 발신 작업은 반드시 사용자 승인.
- **규칙:** 잠금(confirmed🔒)·집행(spent)·계약 항목 불변. 재기획은 **잔여 예산**(전체 − 집행 − 확정) 범위 내 재배분. 숫자는 계산 결과 기반(창작 금지), 인용은 출처 명시.
- 설정: 사고 수준은 `.pi/settings.json`의 `defaultThinkingLevel`(현재 `medium`)을 따른다.

## researcher — 외부 조사 (`researcher.md`)
- **역할:** '바깥세상' 정보(장소·업체·예산 시세·온라인 사례·날씨 등)를 능동적으로 조사.
- **경계:** 조사만. 예산 확정·제안서 작성·발송 등 결정/행동 안 함. 완료 후 main에 보고하고 **즉시 종료(일회성)**.
- **재사용:** 장소/업체/날씨로 쪼개지 않은 "바깥세상 조사"라는 한 역할이라, main뿐 아니라 monitor도 호출해 여러 곳에서 재사용한다.
- **병렬:** 타입은 1개지만 런타임에는 **여러 인스턴스를 동시에** 띄운다 — main이 독립 조사 갈래(venue / 업체+시세 / 온라인 사례 / 날씨)마다 별도 researcher를 병렬 spawn한다. 한 인스턴스가 모든 주제를 순차로 맡지 않는다. (타입을 쪼개지 말라는 원칙과 충돌하지 않음 — 쪼개는 건 *타입*, 병렬은 *인스턴스*)
- **경계 보강:** 내부 과거 사례 DB(`case_search`)는 researcher가 아니라 **main이 직접** 조회한다(researcher = 바깥세상 전담).
- **규칙:** 한국어. 출처 필수, 못 찾으면 "정보를 찾지 못했습니다"로 명확히 보고(창작 금지).
- 설정: `run_in_background: true`. (외부 조사라 extensions는 기본 로드 — `isolated` 아님)

## writer — 제안서 초안 작성 (`writer.md`)
- **역할:** main이 준 조사 결과·예산·업체 정보로 제안서 초안 작성. (프롬프트 본문 호칭은 "Author")
- **구조:** 제목 / 문단 블록 / 확정값 표시 / 출처 인용([1], [2] …).
- **원칙:** 확정 수치는 데이터 그대로(창작 금지), 미정은 '미정'·'확인 필요'로 표기. critic이 검토하기 쉽게 명확한 구조로 출력. 검토·발송·확정은 안 함. 완료 후 부모에 전달·종료.
- **루프:** writer는 critic과 **순차 루프**로 묶인다. critic이 "문제"를 내면 main이 그 피드백을 writer에 재투입해 수정시키고 다시 critic에 검토 — "통과" 또는 최대 2회까지.
- 설정: `tools: read`만, `model: haiku`(fuzzy), `thinking: high`.

## critic — 제안서 검토 (`critic.md`)
- **역할:** 제안서를 다각도로 검토해 문제를 지적하고 통과 여부 판정. **직접 수정 안 함**(수정은 writer/main 몫).
- **점검 항목:** ① 리스크(날씨·안전·업체 취소/지연·일정 충돌) ② 인용·근거 ③ 잠금 위반 ④ 예산 초과 ⑤ 누락 항목.
- **피드백:** 문제 시 "무엇이·왜" 구체적으로, 문제 없으면 "통과". 한국어. 직접 수정은 안 하므로 수정 주체는 늘 writer/main (위 writer §의 루프 참고).
- 설정: `model: sonnet`(fuzzy), `inherit_context: false`.

## secretary — 통신 정리 (`secretary.md`)
- **역할:** 외부 통신 채널(현재 이메일, 추후 Slack 등) 메시지를 읽어 **회의록 형태로 정리**. 읽기는 MCP(Gmail 등) 경유.
- **경계:** 중립적 정리만(사실 추출, `[태그]` 부착). 판단·제안·상태 변경·파일 쓰기 안 함. 회의록은 **Planner(보드 반영 제안)와 monitor(재기획 판단)** 양쪽의 입력이 된다.
- **채널 독립성:** 채널이 바뀌어도 정리 로직 동일, **메시지 읽기 메커니즘만 교체** 가능하게 설계.
- 예: "○○업체 견적 회신 도착, 금액 ○○원.", "인원 250명으로 늘리는 안 논의됨."
- 설정: `tools: read, grep, find, ls`, `extensions: true`. MCP 도구는 `mcp`를 tools에 넣지 않고(빌트인 도구명이 아니라 로드 실패) **`extensions`로 로드해 노출**한다. 현재 `.mcp.json`은 비어 있어 실제 MCP 서버는 추후 등록.

## monitor — 변화 감지·판단 (`monitor.md`)
- **역할:** 워크스페이스의 여러 변화를 모아 "기존 기획을 뒤엎을 만한가, 수정 제안 가치가 있는가" 판단. 필요 시 main에 **재기획 깃발(trigger)**. 직접 수정 안 함. (프롬프트 본문 호칭은 "Watcher")
- **입력:** ① `external_signals`(현재 날씨—우천 확률·기온 등, 추후 교통·주차 등; 주기 점검 후 이전 값과 비교) ② secretary가 정리한 **회의록**(통신에서 오는 변화) ③ **사용자 직접 변경**(값 수정·잠금 토글·업체 단계 이동) ④ 기타 변동 입력.
- **판단:** 변화 영향 평가 후 '수정 제안할 만함 / 무시 가능' 등급화. 잠금(confirmed)·집행(spent)·계약 완료 항목은 판단 대상에서 **제외**.
- **실행:** `schedule`로 일정 간격마다 깨어나 외부 신호 점검. 읽기·판단 위주.
- 설정: `run_in_background: true`, `model: sonnet`(fuzzy), `thinking: high`.

> **"바깥 정보"를 다루는 세 에이전트는 방향이 달라 안 겹친다:**
> researcher = 나가서 *찾아옴* · monitor = *지켜봄* · secretary = 들어온 걸 *받음*.

## 협업 흐름 (요약)

```
사용자 ─조건─▶ event-organizer-main (세션 메인)
   │
   ├ [조사: 병렬 fan-out]  case_search()(TODO) + researcher #1..N (run_in_background)
   │        └─▶ get_subagent_result로 회수 → 종합
   ├ [선택 게이트]  확정될 후보(장소·업체)는 ask_user_question으로 사용자가 고름 → confirmed🔒
   ├ [작성·검토 루프]  writer(초안) ⇄ critic(판정+수정방향+처리주체)  "통과" 또는 ≤2회
   │        └ critic 처리주체대로: writer 재작성 / 재조사(researcher) / 사용자 판단(ask_user_question)
   └ 제안서 완성 / 메일 본문 ──승인 게이트(ask_user_question)──▶ Gmail MCP 발송

[운영 중 변화 대응]
메일·슬랙 ─▶ secretary(회의록·사실 추출) ─┬─▶ Planner: ask_user_question으로 보드 반영 제안 → update_state(TODO)
                                       └─▶ monitor: 재기획 판단
외부 신호(날씨)·사용자 변경 ─schedule─▶ monitor ─「판정: 재기획 제안」─▶ Planner: 계획분만 재기획(잠금/집행/계약 보존)
```

- **조사 병렬:** researcher는 *타입 1개*지만 main이 독립 갈래마다 *인스턴스를 동시에* 띄운다(한 메시지 안에 여러 `Agent` 호출 + 각 `run_in_background`, 동시성 4). 내부 과거 사례 DB(`case_search`)는 main이 직접 조회해 외부 결과와 종합.
- **선택 게이트:** 확정으로 이어지는 후보는 main이 임의로 고르지 않고 `ask_user_question`으로 사용자에게 제시해 고르게 한다(통제 철학: 확정·선택은 사람). 시세·사례·날씨 같은 맥락 정보는 선택 대상 아님.
- **작성·검토 루프:** writer ⇄ critic은 병렬 형제가 아니라 순차 루프. critic이 문제를 **처리 주체(writer/재조사/사용자 판단)로 분류**하면 main이 그대로 라우팅 → writer로 못 고칠 건 writer에 안 돌림(무한 루프 방지). "통과" 또는 최대 2회.
- **변화 대응:** secretary는 **회의록(사실)만** 추출 → main이 `ask_user_question`으로 보드 반영 제안 / monitor가 재기획 판단(`판정: 재기획 제안`이면 계획분만 재배분).
- **AI가 빠르게:** 조사·초안·검토·계산. **사람이 개입:** 확정·수정·발송 등 돈이 들거나 외부로 나가는 모든 작업(승인 게이트).
- **불변 가드:** 잠금(confirmed🔒)·집행(spent)·계약 항목은 어떤 에이전트도 변경하지 않으며, 재기획은 잔여 예산 범위 내에서만.

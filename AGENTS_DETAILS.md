# 행사기획 에이전트 아키텍처

행사 1개 = 워크스페이스 1개 = pi 세션 1개. 한 세션 안에서 아래 에이전트들이 일한다.
홈 화면은 세션이 아니라 여러 세션을 가로질러 보여주는 조회 대시보드 (에이전트 없음).

---

## 1. 워크플로우 (기본 흐름)

```
사용자 조건 입력
  → 정보 충분한가?  ─ 부족하면 물어보기
  → 외부 조사 (Researcher 병렬 fan-out): 업체 · venue · 예산 · 온라인 사례 · 날씨
  + 과거 DB 조회 (Planner가 case_search로 직접)
  + external_signals 수집 (현재: weather 샘플, 확장 가능)
  → 종합 → (선택 게이트: 확정될 후보는 사용자가 고름 `ask_user_question`) → 기획 → 초안(Author) ⇄ 검토(Critic) 루프(≤2회) → 제안서
  ※ 이 전 과정 어디서든 사용자 input 필요하면 그때그때 물어봄
```

기획 이후로 이어지는 전체 루프: 기획 → 제안서 → (사람 확정) → 발송 → 정산 → 사례 DB 적립 → 다음 행사에서 재사용.

## 2. 에이전트 (6)

### Planner — 메인
- 세션의 유일한 메인. 사용자와 대화하는 얼굴.
- 기획 전체를 몰고 감. 되묻기·예산 산정·정산·발송을 직접 처리.
- 조사는 Researcher에게, 작성은 Author에게, 검토는 Critic에게 위임하고 결과를 종합.
- 자기 행사 하나만 책임짐. 홈/다른 행사는 모름.

### Researcher — 외부 조사
- 능동적으로 *찾아오는* 애. 업체·venue·예산 시세·온라인 사례·external_signals.
- 장소/업체/날씨로 안 쪼갬. "바깥세상 알아보는" 한 역할. → 단 이건 **타입**을 안 쪼갠다는 뜻. 런타임엔 Planner가 **독립 갈래마다 인스턴스를 병렬로** 띄운다(한 메시지에 여러 `Agent` 호출 + 각 `run_in_background`, 동시성 4). 한 인스턴스가 모든 주제를 순차로 맡지 않음.
- 내부 과거 사례 DB는 Researcher가 아니라 **Planner가 `case_search`로 직접** 본다 (Researcher = 바깥세상 전담).
- Planner도 쓰고 Watcher도 씀 (다양하게 재사용).

### Watcher — 변화 감지 + 재기획 판단
- 변화를 *지켜보는* 애. "이 변화가 기존 기획을 흔드나?"를 판단.
- 보는 출처:
  - external_signals 변동 (날씨 등) — 긁어와서 비교
  - 사용자 직접 변경 — 값 수정·잠금 토글·업체 단계 이동
- 판단만 함: "이 정도는 재기획 불필요 / 이 정도는 재기획 제안." 직접 안 고침.
- **트리거: 시스템(백엔드)이 스케줄로 실행** — Planner가 spawn하지 않는다. 판정 결과만 세션으로 들어가 Planner가 대응. (⚠️ 트리거 인프라 미구현)
- 재기획이 실제 하는 일은 §4 참고 (집행·확정 보존, 계획분만 재배분).

### Secretary — 통신 해석 (회의록 추출만)
- 들어온 *통신을 받아 처리하는* 애. 메일·슬랙을 읽고 **일어난 사실만 중립적으로 추출**해 회의록으로 정리. `[업체]`·`[예산]` 같은 영향 태그를 붙임. **판단·제안은 안 함.**
- **트리거: 시스템이 통신 도착 시 실행** — Planner가 spawn하지 않는다. 회의록이 세션에 들어가면 Planner가 대응. (⚠️ 트리거 인프라 미구현)
- 추출된 사실은 두 갈래로 나뉜다 (분리형):
  - **Planner**가 보드 반영을 사용자에게 제안 (예: 메일 "견적 보내드립니다" → "견적 단계로 올릴까요?", 슬랙 "결제했습니다" → "계약+집행으로 반영할까요?"). 승인 시 Planner가 `update_state`. 결제·계약처럼 틀리면 안 되는 건 반드시 사람 확인.
  - **monitor**가 같은 회의록으로 재기획 필요성 판단.

### Author — 제안서 초안
- 제안서 초안 작성.
- Critic과 **순차 루프**로 묶임 (아래 Critic 참고). 조사가 병렬이라면 작성·검토는 루프.

### Critic — 다각도 검토
- Author 초안을 여러 각도에서 깐다 (한 체크리스트 훑기가 아니라 다각도).
- 각도: 리스크 탐지 · 인용/근거 · 잠금 위반 · 예산 초과 · 누락 등.
- **루프:** Author 초안 → Critic 검토 → "문제"면 Planner가 그 피드백을 Author에 재투입해 수정 → 다시 Critic. "통과" 또는 **최대 2회**까지. Critic은 직접 안 고치므로 수정 주체는 늘 Author.

> 세 애가 "바깥 정보"를 다루지만 방향이 달라 안 겹침:
> Researcher = 나가서 찾음 · Watcher = 지켜봄 · Secretary = 들어온 걸 받음.

## 3. 보조 구성요소 (에이전트 아님)

### Skill — 절차 문서
- `meeting-notes` — 슬랙 날것 메시지 뭉텅이 → 회의록 형태로 정리. Secretary(또는 Planner)가 필요할 때 호출.
- (그 외 제안서 작성·리스크 점검 등 절차도 skill 후보 — 추후 정리)

### MCP — 외부 연결
- Gmail — 메일 읽기/발송
- Slack — 내부 메시지 읽기 (회의록 원천)
- 날씨 API — external_signals (현재 샘플)
- 지도 — venue·동선
- 검색 — 외부 조사

### update_state — 쓰기 통로
- 모든 state 변경이 이걸 통해서만 일어남. Planner·Watcher·Secretary 등 누구나 이걸로만 보드를 고침.
- 안에 잠금/집행 가드(hook)가 박혀 있음 (§4).

### Hook — 쓰기 가로채기 (가드)
- update_state로 들어오는 쓰기를 가로채 강제 검사:
  - **잠금 가드** — locked / confirmed🔒 항목이면 차단 (사람이 정한 것 보호)
  - **집행 가드** — 이미 집행/계약된 것이면 차단 (현실에서 일어나 못 되돌림)
  - **승인 게이트** — 발송·적립 직전, 사람 승인 없으면 차단
- LLM에 "하지 마" 부탁이 아니라 코드가 물리적으로 막음.

## 4. 상태 규칙 (재기획의 핵심)

변화가 들어와도 **이미 정해졌거나 실행된 것은 안 건드린다.** 예산·업체 둘 다 같은 논리.

| | 자유 (재기획 대상) | 잠금 (사람이 정함) | 실행됨 (못 되돌림) |
|---|---|---|---|
| 예산 | 계획(planned) | 확정🔒(confirmed) | 집행(spent) |
| 업체 | 후보 · 견적 | 확정🔒 | 계약 |

- **업체 "계약" = 예산 "집행" 발생** (둘이 연동. 케이터링 계약 → 케이터링 예산이 집행분으로).
- 재기획 = `남은 돈(전체 − 집행 − 확정)` 안에서 **계획분만** 다시 배분.
  - 집행분 → 출발선에서 빠짐 (되돌리는 게 아니라 애초에 제외)
  - 확정분 → 그대로 (잠금)
  - 계획분 → 남은 돈으로 재배분
- 재기획 후: stale 해제, 버전 +1.

---

## 5. 변화의 세 출처 (Watcher·Secretary가 나눠 봄)

1. **외부 신호 변경** — external_signals(날씨 등). → Watcher가 긁어와 감지.
2. **사용자 직접 변경** — 값 수정·잠금 토글·업체 단계 이동. → 즉시 감지.
3. **통신에서 오는 변화** — 메일·슬랙. → Secretary가 회의록으로 사실만 추출 → Planner가 "반영할까요?" 제안(`ask_user_question`) / monitor가 재기획 판단.

---

## 6. external_signals — 확장형 칸

외부 API로 들어오는 변동 데이터 칸. 날씨 전용이 아니라 확장 가능한 구조.
- 현재 구현: `weather` (샘플)
- 확장 후보: 교통/대중교통 · 주차 · 인근 경합행사 · 가격 시세
- 코드는 `external_signals.weather` 식으로 짜서 나중에 `.traffic` 등 추가 가능하게.
- 지금 다 붙일 필요 없음 — 날씨만 확실히, 나머지는 자리만.
